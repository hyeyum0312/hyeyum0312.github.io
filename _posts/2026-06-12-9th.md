## 들어가며

가족 캘린더 서비스에서는 데이터를 자주 새로 불러오도록 React Query를 설정해뒀습니다.

```js
new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 0,                 // 받아온 데이터를 곧바로 "오래된 것"으로 취급
      refetchOnWindowFocus: true,   // 창에 다시 포커스가 오면 새로 불러옴
      refetchOnMount: true,         // 화면에 다시 들어와도 새로 불러옴
    },
  },
});
```

데이터를 신선하게 유지하는 데는 좋지만, 편집 화면에서는 문제가 될 수 있는 설정입니다.

사용자가 일정을 수정하는 중에 데이터가 새로 불러와져 화면을 갈아끼우면, 입력하던 값이 사라질 수 있기 때문입니다.

비동기 화면에서 흔히 말하는 "뒤늦게 도착한 데이터가 내 입력을 덮어쓰는" 문제입니다.

그런데 동작을 따라가 보니, 이렇게 공격적으로 새로고침을 돌리는데도 **편집 폼은 멀쩡했습니다.**

이 글은 버그를 고친 이야기가 아니라, **왜 이런 설정에서도 편집 폼이 안 깨졌는지**를 정리한 글입니다.

결론부터 말하면, 편집하는 동안에는 서버 데이터가 화면으로 들어올 길이 막혀 있었기 때문입니다.

---

## 입력이 덮어써지려면 사실 여러 단계를 거쳐야 한다

"새로고침이 내 입력을 덮어쓴다"는 일이 실제로 일어나려면, 사실 아래 단계들이 차례로 다 맞아떨어져야 합니다.

```text
백그라운드에서 데이터 새로고침
   ↓
서버 데이터(allSchedules)가 갱신됨
   ↓
화면이 들고 있던 "지금 선택된 일정"이 바뀜   ← (1) 여기가 출발점
   ↓
편집 폼의 초기값이 바뀜
   ↓
폼이 초기화되면서 입력하던 값이 사라짐
```

여기서 핵심은 **(1)만 안 일어나면 그 뒤는 자연히 안 일어난다**는 점입니다.

그래서 "왜 안 깨졌을까"를 확인하는 기준도 (1)에 맞췄습니다. 새로고침이 정말 (1)을 일으킬 수 있는지만 보면 됐기 때문입니다.

이 서비스에는 편집이 일어나는 화면이 두 종류 있었는데, 확인해보니 두 화면 모두 (1)이 일어나지 않는 구조였습니다. 각각 이유는 달랐습니다.

---

## 첫 번째 화면 — AI가 뽑아준 일정 편집하기

사진이나 음성을 올리면 AI가 일정을 알아서 뽑아주는 기능이 있습니다.

흐름은 이렇습니다.

```text
사진 또는 음성 업로드
       ↓
AI가 일정 추출 (시간이 걸리는 요청)
       ↓   ← 여기서 요청이 완전히 끝남
편집 화면으로 이동
       ↓
사용자가 편집
       ↓
한 번에 등록
```

여기서 눈여겨볼 점이 두 가지입니다.

**하나, 편집 화면에 들어올 때는 이미 AI 작업이 끝나 있습니다.**

```text
AI 분석 완료 → navigate(편집 화면, { state: 뽑아낸_일정들 })
```

분석이 끝난 결과를 들고 화면으로 넘어오기 때문에, 편집하는 도중에 "뒤늦게 도착할 AI 응답" 같은 건 애초에 없습니다.

**둘, 편집하는 동안에는 서버를 다시 부르지 않습니다.**

오직 화면 안에 들고 있는 로컬 데이터(배열)만 가지고 편집합니다.

```text
AI 결과
   ↓
화면 안에 로컬로 보관
   ↓
사용자가 편집 (이때는 서버를 부르지 않음)
   ↓
등록 버튼을 누를 때 한 번에 전송
```

편집하는 동안 서버 데이터가 끼어들 틈이 없으니, 앞에서 본 (1)이 생기질 않습니다.

여기서 한 가지 짚어둘 점은, 애초에 서버를 부를 이유 자체가 없었다는 것입니다. AI가 추출한 일정은 사용자가 [등록]을 누르기 전까지는 서버에 저장된 데이터가 아니라 "등록 후보"일 뿐입니다. 서버를 조회해도 이 추출 결과는 나오지 않습니다. 즉 로컬을 쓴 건 성능을 위한 선택이라기보다, 가져올 대상이 아직 서버에 없는 "초안" 단계의 데이터였기 때문입니다.

추가로, 편집은 받은 데이터를 그 자리에서 직접 바꾸는 대신 불변 업데이트 방식으로 처리했습니다. 값을 고칠 때마다 기존 배열과 객체를 건드리지 않고, 새 배열에 바뀐 항목만 새 객체로 교체합니다.

```text
편집 전: [ A, B, C ]
   ↓  B의 제목 수정
편집 후: [ A, B', C ]   ← 새 배열, B만 새 객체로 교체
```

이렇게 하면 React가 "무엇이 바뀌었는지"를 참조 비교만으로 구분할 수 있습니다.

배열 참조가 바뀌었으니 화면이 다시 그려지고, 리스트를 그룹핑하는 `useMemo([schedules])`도 다시 계산됩니다. 그 안에서도 새 객체로 교체된 `B`만 다시 그려지고, 참조가 그대로인 `A`·`C`는 건너뜁니다.

반대로 기존 객체를 그 자리에서 고쳤다면 참조가 그대로라, React는 무엇이 바뀌었는지 알아채지 못했을 겁니다. 불변 업데이트가 중요한 건 이 때문입니다.

대신 **등록이 끝난 다음에는** 로컬 데이터를 계속 붙들고 있지 않았습니다. 쿼리를 무효화(invalidate)하고 홈으로 돌아가면, 홈 화면이 다시 마운트되면서 서버 데이터를 새로 불러옵니다.

등록하고, 수정하고, 실패하고, 다시 시도하고… 이런 상황이 섞이면 화면 상태와 서버 응답을 직접 맞춰주는 쪽이 오히려 더 헷갈리기 때문입니다.

정리하면 이렇게 선을 그은 셈입니다.

> 편집하는 동안에는 사용자의 입력만 믿는다.
>
> 등록이 끝나면 다시 서버를 믿는다.

## 두 번째 화면 — 캘린더에서 기존 일정 수정하기

AI 편집과 달리, 캘린더에서 기존 일정을 수정하는 화면은 서버 데이터와 더 가까이 붙어 있습니다.

이 화면에는 서버 데이터가 바뀌면 그걸 화면에 반영해주는 코드가 있습니다.

핵심만 추리면 이렇습니다.

```js
useEffect(() => {
  // 상세 시트가 닫혀 있으면 아무것도 안 하고 그냥 끝냄
  if (!isDetailSheetOpen || !selectedSchedule) return;

  const updated = allSchedules.find((s) => s.id === selectedSchedule.id);
  if (updated && updated !== selectedSchedule) {
    setSelectedSchedule(updated); // 새로고침된 내용을 상세 화면에 반영
  }
}, [allSchedules, isDetailSheetOpen, selectedSchedule]);
```

언뜻 보면 위험해 보입니다. 새로고침으로 `allSchedules`가 바뀌면 `selectedSchedule`도 바뀌고, 그게 편집 폼으로 넘어가서 입력을 지울 것 같기 때문입니다.

하지만 포인트는 **이 코드가 첫 줄에서 막혀 있다**는 점입니다.

`isDetailSheetOpen`이 `false`면, 그 아래는 아예 실행되지 않습니다.

그리고 수정 화면을 열 때는 **반드시 상세 시트를 먼저 닫습니다.**

```js
// 일정 수정으로 들어갈 때
setIsDetailSheetOpen(false); // 상세 시트를 먼저 닫고
setIsEditSheetOpen(true);    // 그다음 수정 시트를 연다
```

그래서 수정 폼이 떠 있는 동안에는 항상 `isDetailSheetOpen`이 `false`이고, 위의 반영 코드는 **잠들어 있습니다.**

덕분에 수정하는 동안에는 선택된 일정이 처음 값 그대로 멈춰 있고, 폼의 초기값도 안 바뀌고, 폼이 초기화될 일도 없습니다.

새로고침이 `selectedSchedule`을 바꿀 수 있는 길은 이 코드 한 줄뿐이었고(나머지 갱신은 전부 사용자 액션이었습니다), 그 한 줄이 "상세 닫고 수정 연다"는 흐름 때문에 닫혀 있었습니다.

| 상황 | 동작 |
| --- | --- |
| 상세 시트가 열려 있을 때 | 새로고침 내용이 상세 화면에 자연스럽게 반영됨 (원래 의도한 동작) |
| 수정 폼이 열려 있을 때 | 상세 시트가 닫혀 있어 반영 코드가 멈춰 있음 → 폼은 안전 |

단순한 화면 전환처럼 보이는 이 순서가, 결과적으로 새로고침으로부터 편집 폼을 지켜주고 있는 셈입니다.

## 그래도 끝까지 남는 위험 — 앱이 강제로 꺼질 때

이렇게 두 화면 모두 "새로고침이 입력을 덮어쓰는" 문제는 일어나지 않습니다.

하지만 성격이 전혀 다른 위험이 하나 남습니다.

바로 **앱이 강제로 종료되는 상황**입니다.

이 서비스는 Flutter WebView 위에서 돌아가는 구조라, 메모리가 부족하면 OS가 앱을 그냥 꺼버릴 수 있습니다.

실제 테스트 중에도 작업하던 화면이 통째로 사라지는 일을 여러 번 겪었습니다.

이건 값이 "덮어써지는" 문제가 아니라, 작업이 "통째로 날아가는" 문제라 결이 완전히 다릅니다.

서버와 분리해두든 시트를 따로 열든, 앱 자체가 죽으면 화면이 들고 있던 데이터는 같이 사라지니까요.

---

## 어디까지 되살리고, 어디는 안 되살릴지 정하기

그래서 앱이 꺼지는 상황에 대비해, **사용자가 어떤 화면에서 작업 중이었는지**는 다시 살려주도록 했습니다.

예를 들어 일정 등록 시트를 열어둔 상태였다면, 앱을 다시 켰을 때 그 시트를 다시 열어줍니다.

다만 **편집하던 일정 데이터 자체는 되살리지 않았습니다.**

이건 일부러 그렇게 둔 선택이자, 솔직히 이 화면의 한계이기도 합니다.

편집하던 값을 자동으로 되살려주면, 사용자가 그걸 "이미 저장된 값"이라고 착각할 수 있다고 봤기 때문입니다.

되살린 값이 실제 저장된 값과 다를 때 생기는 혼란이, 다시 입력하는 번거로움보다 더 위험하다고 판단했습니다.

그래서 **화면 상태(어떤 시트가 열려 있었는지)는 되살리되, 편집 데이터는 되살리지 않는** 선을 그었습니다.

## 마무리

`staleTime: 0`에 `refetchOnWindowFocus`까지 켜둔, 데이터를 꽤 자주 새로 불러오는 설정이었습니다.

그런데도 편집 폼이 안 깨진 이유는, 편집하는 동안에는 서버 데이터가 화면으로 들어올 길이 막혀 있었기 때문입니다.

- **AI 일정 편집**은 로컬 데이터로만 다뤄서 서버가 끼어들 틈이 없었고,
- **캘린더 일정 수정**은 "상세 닫고 수정 연다"는 순서 덕분에 반영 코드가 잠들어 있었습니다.

두 경우 모두, 충돌이 나면 처리한 게 아니라 **충돌이 시작되는 첫 단계가 아예 일어나지 않았던** 셈입니다.

그리고 마지막까지 남는 강제 종료에 대해서는, 화면 상태는 되살리되 편집 데이터는 되살리지 않는 선을 일부러 그었습니다.

결국 비동기 작업이 많은 화면에서 중요한 건, 데이터를 얼마나 신선하게 유지하느냐가 아니라

**언제 어떤 데이터를 믿을지, 어떤 길은 일부러 막아둘지, 그리고 어떤 위험은 굳이 다루지 않을지를 분명히 정해두는 것**이라는 점을 배울 수 있었습니다.
