# 🔍 들어가며

두 서비스를 만들면서 가장 손이 많이 간 건 정상 동작이 아니라 "끊겼을 때"였습니다.

가족 캘린더와 의료 STT, 둘 다 AI 처리가 오래 걸리는 기능이 있었고, 사용자는 처리가 끝나기 전에 화면을 벗어나거나 앱을 닫을 여지가 있었습니다. 거기에 토큰이 중간에 만료되거나 네트워크가 잠깐 끊기는 일까지 겹치면, 정상 흐름만 따라가서는 잘 보이지 않던 문제들이 한꺼번에 드러났습니다.

이 글은 토큰이 만료될 때, 네트워크가 끊길 때, AI 처리가 오래 걸리는 동안 사용자가 떠날 때, 이 세 가지를 두 서비스에서 어떻게 다뤘는지 돌아본 기록입니다. 한쪽은 Flutter WebView 위에 React를 얹은 가족 캘린더, 다른 한쪽은 의료용 STT 웹이라 구현은 꽤 달랐는데, 끝나고 복구 코드를 모아보니 제가 내린 결정들이 결국 비슷한 곳을 향하고 있었습니다.

---

# 🔍 토큰이 만료됐을 때

처음엔 단순하게 생각했습니다. 401 떨어지면 refresh 한 번 하고 다시 보내면 되지 않나.

그런데 만들다 보니 그 앞에 한 겹을 더 두게 됐습니다. 요청을 보내기 전에 토큰을 한 번 까보는 것이었습니다. 가족 캘린더에서는 만료 2분 전이면 미리 갱신을 걸어두되 현재 요청은 그대로 보냈고, 의료 STT에서는 이미 만료된 토큰이면 헤더를 아예 안 붙였습니다. 어차피 실패할 요청을 굳이 서버까지 보내고 싶지 않았습니다. 그래도 놓친 건 서버가 알려줬습니다. 가족 캘린더는 401로, 의료 STT는 응답 속 에러코드로 토큰 문제를 잡아 갱신에 들어갔습니다.

까다로웠던 건 토큰이 만료되는 순간 여러 요청이 동시에 깨질 때였습니다. 가족 캘린더에서는 갱신 중에 들어온 요청을 큐에 쌓아뒀다가, 새 토큰이 나오면 한꺼번에 다시 보내도록 묶었습니다. 갱신이 한 번만 일어나도록. 의료 STT는 실패한 요청이 저마다 갱신을 부르다 보니, 동시에 만료되면 갱신 요청이 중복으로 나갈 수 있습니다. 가족 캘린더처럼 갱신을 한 번으로 묶는 건 다음 숙제로 남겨뒀습니다. 사실 가족 캘린더의 큐도 처음부터 있던 게 아니라, 한 번 부딪히고 나서 넣은 것이었습니다.

```text
요청 A ─┐
요청 B ─┼─ (동시에 401)
요청 C ─┘
        │
        ▼
   이미 갱신 중? ──아니오──► 갱신 1회 시작
     │ (예)                     │
     ▼                          ▼
   큐에서 대기 ◄───────────  새 토큰 발급
        │
        ▼
   대기하던 요청 전부 새 토큰 달고 재시도
```

그리고 또 다른 에러를 발견했습니다. 의료 STT에서 파일 업로드 요청을 그대로 재시도했더니 multipart boundary가 깨졌습니다. 그 뒤로는 재시도 전에 FormData를 복제하고 Content-Type을 비워 boundary가 새로 잡히게 했습니다.

갱신까지 실패하면 토큰을 지우고 로그인 화면으로 보냈습니다. 멈춰 있는 빈 화면만큼은 남기고 싶지 않았습니다.

# 🔍 네트워크가 끊겼을 때

이건 두 서비스에서 다르게 갔는데, 일부러 그랬다기보단 상황이 달라서 자연스럽게 갈렸습니다.

가족 캘린더에서는 연결이 끊기면 전체화면 오버레이를 띄우고(순간 끊김은 2초쯤 무시), 다시 연결되면 화면 캐시를 믿지 않고 서버에 다시 물어봤습니다. 재시도는 결과가 바뀔 요청만 했습니다. 4xx처럼 다시 보내도 답이 같은 건 건너뛰었습니다.

의료 STT에서는 일반 HTTP에 전역 감지를 두지 않았습니다. 네트워크 에러는 화면별로 받게 두고, 폴링은 자동 재시도를 끈 채 다음 폴링에서 회복하게, 녹음 업로드는 사용자가 다시 누르게 했습니다. 전역 상태를 들고 다니기보다 끊긴 자리에서 그때그때 처리하는 쪽이 단순했습니다.

---

# 🔍 AI가 오래 걸리는 동안 사용자가 떠났을 때

세 가지 중 제일 신경 썼던 부분입니다. AI 처리는 오래 걸리고 사용자는 그새 떠나니, 결정은 분명했습니다. 진행 상태를 클라이언트가 들고 있지 않기로 했습니다.

가족 캘린더의 이미지 일정 추출은 업로드하고 Job을 만든 다음 jobId만 받았습니다. 무거운 일은 서버가 하고, 화면은 jobId로 결과를 폴링해 가져왔습니다.

의료 STT의 AI 요약도 같은 식이었습니다. 요약을 걸면 서버가 녹음을 처리 중 상태로 두고, 앱이 죽어도 서버는 계속 돌렸습니다. 다시 들어오면 서버 목록을 새로 받아 처리 중인 항목을 골라 폴링을 이어갔습니다.

"지금 뭘 처리 중"인지를 메모리에 들고 있는 게 아니라, 그때그때 서버에 물어봐서 알게 한 것입니다.

```text
[클라] 업로드 → Job 생성 요청
                  │
                  ▼
[서버] Job 처리 시작 ───► 사용자가 떠나도 계속 처리
                  │
        ┌─────────┘
        ▼
[클라] jobId 받음 (메모리에만 보관)
        │
   (이탈 / 새로고침 / 앱 종료)
        │
        ▼
[클라] 재진입 → "처리 중인 거 있어?" 서버에 다시 질의
        │
        ▼
   서버 목록에서 '처리 중' 발견 → 폴링 재개 → 결과 회수
```

다만 가족 캘린더는 Job 상태를 메모리에만 들고 있어서, 잠깐 화면을 벗어났다 오는 건 폴링으로 회수됐지만 앱이 완전히 죽었다 켜지면 jobId가 사라졌습니다. jobId를 IndexedDB처럼 브라우저에 저장해 앱이 꺼져도 남게 했다면 하드 재시작에도 버텼겠지만, 그때는 이후 Flutter 네이티브 앱으로 전환하는 방향을 염두에 두고 있어서 웹에 그렇게까지 저장해두진 않았습니다. 그래서 이 경우 복구는 네이티브가 jobId를 다시 넘겨주느냐에 달려 있고, 서버 목록 자체를 기준으로 삼은 의료 STT가 이 점에선 더 단단했습니다.

# 🧐 정리

따로 만든 두 서비스인데, 끝나고 복구 코드를 나란히 놓으니 제가 내린 결정들이 같은 말을 하고 있었습니다.

> 클라이언트는 자기 메모리를 믿지 않는다. 진행 중인 건 서버에 다시 묻고, 기준은 항상 서버에 둔다.

토큰은 만료를 짐작하지 않고 서버에 다시 받았고, 네트워크가 돌아오면 캐시 대신 서버에 다시 물었고, AI 작업은 결과를 들고 있지 않고 서버 상태를 폴링해 가져왔습니다. 강제 종료조차 화면을 새로 그리는 대신 저장해둔 상태를 되돌렸습니다.

그리고 복구를 자동으로 시도하되 끝내 안 되면 사용자에게 분명히 넘겼습니다. 의료 STT에서는 실패한 작업에 재시도 버튼을 띄워 사용자가 직접 다시 시도하게 했고, 토큰 갱신이 끝내 실패하면 재로그인으로 보냈습니다. 의료 현장에선 에러 그 자체보다, 실패했는데 사용자가 그걸 모르고 넘어가는 상황이 더 위험하다고 봤기 때문입니다.
