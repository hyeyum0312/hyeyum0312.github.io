# 🔍 들어가며

음성 기록 서비스를 개발하면서 녹음 기능에 필요한 마이크 권한 처리를 맡게 되었습니다.

처음에는 Flutter의 `permission_handler` 라이브러리로 권한 상태를 확인하고 요청하기만 하면 될 거라고 생각했습니다.

하지만 실제 테스트에서 Android 권한 정책은 생각보다 단순하지 않았습니다. 같은 Android 버전이라도 제조사나 기기 환경에 따라 권한 상태가 다르게 동작하는 사례를 여러 번 마주쳤습니다.

이번 글에서는 그 과정에서 만났던 문제들과, 결국 어떤 방향으로 권한 처리를 설계하게 되었는지를 정리해보려고 합니다.

---

# 🔍 Android 마이크 권한

Android 11 이상에서는 마이크 권한 요청 시 여러 가지 선택지가 존재합니다.

- 앱 사용 중에만 허용
- 이번만 허용
- 허용 안 함

사용자는 단순히 권한을 허용하거나 거부한다고 생각할 수 있지만 실제 Android 내부에서는 조금 더 복잡하게 동작합니다.

![Android Permission Options](/images/blogImg/android-permission-dialog.png)

_Android 11 이상 마이크 권한 옵션_

---

# 🔍 실제로 어떤 문제가 있었을까?

처음 구현은 단순했습니다. 사용자가 녹음 버튼을 누르면 마이크 권한을 확인한 뒤 녹음 화면으로 진입하는 방식이었습니다.

```dart
final status = await Permission.microphone.status;

if (status.isGranted) {
  startRecording();
}
```

처음에는 권한 상태만 확인하면 충분할 것이라고 생각했습니다.

하지만 실제 테스트 과정에서 일부 Android 기기에서 예상과 다른 결과가 발생했습니다.

예를 들어

- 사용자가 권한을 거부하지 않았는데 `permanentlyDenied` 로 인식되는 경우
- "이번만 허용"을 선택했는데도 권한이 유지되는 경우
- 동일한 Android 버전임에도 제조사별로 다른 결과가 반환되는 경우

등의 문제가 발생했습니다.

# 🔍 문제 1. 같은 Android인데 상태가 다르게 반환된다

테스트 중 일부 Android 13 기기에서

"이번만 허용"

권한을 선택했음에도 아래 상태가 반환되는 경우가 있었습니다.

```dart
PermissionStatus.permanentlyDenied
```

반대로 어떤 기기에서는

```dart
PermissionStatus.granted
```

로 반환되는 경우도 있었습니다.

처음에는 구현 문제라고 생각했지만 조사 과정에서 permission_handler GitHub 이슈에서도 동일한 사례를 발견할 수 있었습니다.

- [permission_handler GitHub Issue #411](https://github.com/Baseflow/flutter-permission-handler/issues/411)
- [permission_handler GitHub Issue #1145](https://github.com/Baseflow/flutter-permission-handler/issues/1145)

## 왜 이런 현상이 발생할까?

핵심은 **Android 11(API 30)에서 권한 정책이 바뀐 것**에 있었습니다.

Android 11은 권한과 관련해 크게 두 가지를 바꿨습니다.

- **"이번만 허용(one-time)" 도입** — 그리고 앱이 백그라운드로 전환되거나 프로세스가 종료되면 권한이 **자동으로 해제**됩니다.
- **"다시 묻지 않음"의 암묵적 적용** — 별도 체크박스 없이, 사용자가 권한을 **두 번 거부하면 영구 거부**로 취급됩니다.

문제는 permission_handler가 Android의 상태를 판정할 때 `shouldShowRequestPermissionRationale()` 값에 기대고 있다는 점입니다.

- 권한 없음 + `shouldShowRequestPermissionRationale == true` → `denied`
- 권한 없음 + `shouldShowRequestPermissionRationale == false` → `permanentlyDenied`

그런데 `shouldShowRequestPermissionRationale`가 `false`가 되는 상황은 한 가지가 아닙니다.

| 실제 상황                       | shouldShowRationale | 판정              |
| ------------------------------- | ------------------- | ----------------- |
| "이번만 허용" 후 **자동 해제**  | false               | permanentlyDenied |
| 두 번 거부(**진짜 영구 거부**)  | false               | permanentlyDenied |

즉 Android 11이 "one-time 해제"라는 **새로운 상태**를 만들었는데, 이 상태가 기존의 "영구 거부"와 똑같이 `shouldShowRationale == false`로 나오기 때문에, permission_handler가 **둘을 구분하지 못하고 모두 `permanentlyDenied`로 판정**해버리는 것이었습니다.

사용자는 "다시 묻지 않음"을 누른 적이 없는데도 영구 거부로 인식되니, 앱이 잘못 설정 화면으로 안내하게 됩니다.

결국

> Flutter가 반환하는 상태값만으로 사용자 흐름을 제어하면, Android 11이 늘려놓은 상태 변화를 놓쳐 예상치 못한 문제가 발생할 수 있다.

는 점을 확인할 수 있었습니다.

# 🔍 문제 2. "이번만 허용"인데 권한이 계속 남아있다

또 다른 문제는 권한 해제 시점이었습니다.

처음에는

> 앱 종료 = 권한 제거

라고 생각했습니다.

### 공식 문서가 말하는 동작

Android 공식 문서가 설명하는 "이번만 허용(one-time)" 권한의 해제 시점을 요약하면 이렇습니다.

화면에 떠 있는 액티비티(visible activity)나 포그라운드 서비스가 모두 사라지면, 즉 **앱이 백그라운드로 전환되면 짧은 지연 후 권한이 자동으로 해제**됩니다. 물론 프로세스가 종료될 때도 해제됩니다.

즉 "이번만 허용"은 프로세스가 죽어야만 해제되는 것이 아니라, **백그라운드로 전환되기만 해도 곧 해제되는 것이 정상 동작**입니다.

이 점이 일반 런타임 권한("앱 사용 중에만 허용")과 다릅니다. 일반 권한은 백그라운드로 가도 유지되지만, "이번만 허용"은 그렇지 않습니다.

```text
[앱 사용 중에만 허용]        [이번만 허용]
홈 버튼                      홈 버튼
 ↓                           ↓
onPause → onStop             onPause → onStop
 ↓                           ↓
프로세스 유지                백그라운드 전환 감지
 ↓                           ↓
권한 유지                    짧은 지연 후 권한 해제
```

### 그런데 실제 기기는 달랐다

문제는 실제 테스트 결과가 위 공식 동작과 일치하지 않는 경우가 있었다는 점입니다.

| 동작         | 관찰된 권한 유지 여부 | 비고                                      |
| ------------ | --------------------- | ----------------------------------------- |
| 홈 버튼      | 기기마다 다름         | 공식상 해제되어야 하나 유지되는 사례 존재 |
| 뒤로가기     | 기기마다 다름         | 액티비티 finish 여부에 따라 달라짐        |
| 최근 앱 제거 | 해제                  | 프로세스 종료                             |
| 시스템 종료  | 해제                  | 프로세스 종료                             |

특히 일부 삼성 기기에서는 공식 동작과 달리 "이번만 허용" 권한이 백그라운드 전환 후에도 생각보다 오래 유지되는 현상을 확인할 수 있었습니다.

이는 포그라운드 서비스 보유 여부, 제조사별 커스터마이징, OS 버전 차이 등이 복합적으로 작용한 결과로 추정됩니다. 결국 **해제 시점을 코드에서 단정하지 않고, 매 사용 시점에 권한을 다시 확인하는 방향**이 안전했습니다.

# 🔍 iOS는 왜 달랐을까?

Android 권한 이슈를 조사하면서 iOS와의 차이도 함께 확인할 수 있었습니다.

Android는 제조사별 커스터마이징과 OS 버전 차이로 인해 동일한 권한이라도 기기마다 다르게 동작할 수 있습니다.

반면 iOS는 Apple이 하드웨어와 OS를 함께 관리하기 때문에 권한 동작이 비교적 일관적입니다.

예를 들어 마이크 권한의 경우

- 허용
- 거부

상태가 명확하게 구분됩니다.

또한 한 번 권한을 거부하면 앱에서 다시 시스템 권한 팝업을 띄울 수 없으며, 사용자가 직접 설정 화면에서 변경해야 합니다.

Android에서는 제조사별 예외 상황과 권한 상태 변화까지 고려해야 했지만, iOS는 상대적으로 단순한 흐름으로 처리할 수 있었습니다.

> 아래 비교는 **마이크 권한 기준**입니다. iOS도 위치 권한에는 "한 번 허용(Allow Once)"이 존재하지만, 마이크 권한에는 해당 옵션이 없습니다.

| 항목                | Android           | iOS             |
| ------------------- | ----------------- | --------------- |
| 제조사별 차이       | 있음              | 거의 없음       |
| 이번만 허용         | 있음              | 없음            |
| 권한 상태 예외      | 존재              | 상대적으로 적음 |
| 권한 거부 후 재요청 | 일부 가능         | 불가능          |
| 설정 이동 필요      | permanentlyDenied | 권한 거부 시    |

# 🔍 그래서 어떻게 처리했을까?

권한 상태값만 신뢰하지 않도록 구조를 수정했습니다.

---

## 1. 실제 녹음 시점에 권한 확인

앱 실행 시 미리 권한을 요청하지 않고 녹음 버튼을 눌렀을 때 권한을 확인하도록 변경했습니다.

```dart
final status = await Permission.microphone.status;

if (!status.isGranted) {
  await Permission.microphone.request();
}
```

사용자가 왜 권한이 필요한지 이해하기 쉬웠고 불필요한 권한 요청도 줄일 수 있었습니다.

---

## 2. permanentlyDenied 처리 분리

```dart
if (status.isPermanentlyDenied) {
  openAppSettings();
  return;
}
```

권한 요청이 불가능한 상태에서는 반복적으로 시스템 팝업을 띄우지 않고 설정 화면으로 이동하도록 처리했습니다.

---

## 3. 사용자 흐름 기준으로 설계

앞서 정리한 1·2번을 합쳐, 권한을 확인한 뒤 상태에 따라 갈라지는 하나의 흐름으로 정리했습니다. `granted`면 바로 녹음을 시작하고, `denied`면 다시 요청하고, `permanentlyDenied`면 설정 화면으로 안내하는 식입니다.

이렇게 하면서 가장 크게 바뀐 건 관점이었습니다. 권한 상태값 자체보다

> "사용자가 지금 무엇을 하려고 하는가"

를 기준으로 흐름을 설계하게 되었습니다.

# 🔍 다른 접근도 있었다 — 시작 시 일괄 요청 + 미리 안내

위와 같이 "사용 시점에 확인"하는 방식으로 처리한 뒤, 이후 다른 프로젝트에서 권한 처리를 다르게 구현한 사례를 접하게 되었습니다.

그 프로젝트에서는 앱 첫 진입 시 권한을 **한 번에 일괄 요청**하는 방식이었습니다.

```text
앱 첫 진입
       ↓
안내 팝업 1회 (각 권한이 왜 필요한지 설명)
       ↓
"다음" 클릭
       ↓
시스템 권한 팝업 순차 노출
(알림 → 카메라 → 사진 → 마이크)
```

여기서 눈에 띈 점은, 시스템 팝업을 곧바로 띄우는 게 아니라 **그 전에 안내 팝업을 한 번 보여준다**는 것이었습니다.

시스템 권한 팝업은 한 번 거부하면 다시 띄우기 어렵기 때문에, 그 전에 "왜 이 권한이 필요한지"를 먼저 설명해두면 사용자가 무턱대고 거부하는 일을 줄일 수 있습니다.

## 두 방식의 차이

| 기준                | 사용 시점 확인 | 시작 시 일괄 요청      |
| ------------------- | -------------- | ---------------------- |
| 가이드라인 부합     | 권장           | 선택 권한에는 비권장   |
| 허용률              | 높은 편        | 낮아지기 쉬움          |
| 미리 안내 화면      | 없음           | 있음                   |
| 선택 기능이 많을 때 | 적합           | 과한 요청이 될 수 있음 |

애플과 안드로이드 공식 가이드 모두 **권한은 실제로 필요한 순간에, 왜 필요한지와 함께 요청**할 것을 권장합니다.

사용자가 기능을 쓰려는 순간에 요청하면 왜 권한이 필요한지 바로 이해할 수 있어 허용률이 높고, 영문도 모른 채 거부해서 `permanentlyDenied`로 빠지는 상황도 줄일 수 있습니다.

반대로 시작 시 일괄 요청은 사용자가 그 기능을 써보기도 전에 거부할 수 있어, 정작 나중에 기능을 쓰려 할 때 이미 막혀 있는 경우가 생깁니다.

다만 일괄 요청 방식도 **시스템 팝업 전에 미리 안내 화면을 보여준다**는 좋은 점이 있었고, 권한이 앱의 핵심 기능에 꼭 필요한 경우라면 처음에 한 번에 받는 흐름이 더 자연스러울 수도 있습니다.

## 결국 이상적인 형태는

두 방식의 장점을 합치는 것이 가장 좋다고 느꼈습니다.

> 요청 **시점**은 기능을 쓰려는 순간으로 (사용 시점 확인),
> 시스템 팝업을 띄우기 **직전에는 안내 화면**을 한 번 보여주는 것

즉 "언제 요청할지"는 사용 시점 확인 방식을, "어떻게 요청할지"는 사전 안내 방식을 결합하는 것이 사용자 경험 측면에서 가장 좋은 형태라고 생각하게 되었습니다.

# 🧐 정리

이번 작업을 하면서, 권한 처리는 라이브러리 사용법만 안다고 끝나는 문제가 아니라는 걸 느꼈습니다.

특히 Flutter처럼 크로스 플랫폼 환경에서는

- Android 정책
- 제조사별 구현 차이
- 라이브러리의 상태 추상화

이 세 가지가 맞물려 있어, 어느 하나만 봐서는 안 되는 경우가 많았습니다.

결국 가장 중요했던 것은

> permission_handler가 반환하는 상태값보다, 실제 사용자의 경험을 기준으로 흐름을 설계하는 것

이었습니다. 권한 상태를 "신뢰할 값"이 아니라 "참고할 값"으로 바라보기 시작하면서, 기기마다 다르게 동작하던 문제들도 훨씬 다루기 쉬워졌습니다.
