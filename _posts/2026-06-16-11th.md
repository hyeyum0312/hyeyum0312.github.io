# 🔍 들어가며

[마이크 권한 글](/8th/)에서 Android 마이크 권한 하나를 다뤘는데, 실제로 앱에서 챙겨야 할 권한은 마이크만이 아니었습니다.

녹음은 마이크, 문의·이미지 첨부는 갤러리, 앱을 처음 켤 때는 알림. 권한을 확인해야 하는 지점이 화면마다 흩어져 있었습니다. 게다가 같은 권한이라도 iOS와 Android의 동작이 달라서, 화면마다 그 차이를 신경 쓰기 시작하면 끝이 없었습니다.

그래서 권한 처리를 공통 구조 하나로 모았습니다. 앞 글이 "마이크 권한 한 건"이었다면, 이 글은 "권한 전반을 어떻게 추상화했나"에 가깝습니다.

---

# 🔍 권한 체크가 화면마다 흩어져 있었다

권한은 앱을 처음 켤 때 한 번에 요청했습니다. 문제는 그다음이었습니다. 사용자가 그때 거부했을 수도, 나중에 설정에서 꺼버렸을 수도 있으니, 기능을 쓰기 직전에 매번 다시 확인해야 했습니다. 녹음 전에는 마이크가 있는지, 사진을 올리기 전에는 갤러리가 있는지.

이 "권한이 있나? 없으면 다시 요청하고, 영구 거부면 설정으로 안내하고"라는 판단이 화면마다 똑같이 반복됐습니다. 거기에 iOS냐 Android냐에 따른 분기까지 화면 코드에 끼어들면, 정작 그 화면의 본래 기능보다 권한 처리가 더 지저분해졌습니다.

---

# 🔍 호출하는 쪽은 권한 가드 한 블록, 나머지는 훅 안으로

그래서 권한 관련 동작을 `usePermissions` 훅 하나로 모으고, 화면에는 꼭 필요한 API 세 개만 노출했습니다.

- `hasPermission(type)` — 지금 권한이 있는지
- `ensurePermission(type)` — 없으면 요청까지 하고 결과를 boolean으로
- `requestAllPermissions()` — 앱 로드 때 한 번에 요청

호출부는 이런 "권한 가드" 한 블록이 전부입니다.

```ts
if (!hasPermission('microphone')) {
  const permissionGranted = await ensurePermission('microphone');
  if (!permissionGranted) {
    return; // 권한 없으면 동작 중단
  }
}
// 권한 확보 후 본 동작 진행
```

녹음 화면(마이크)도 문의 화면(갤러리)도 글자 그대로 같은 모양입니다. 호출하는 쪽은 **지금 OS가 뭔지, 사용자가 권한을 몇 번 거부했는지, 설정으로 보내야 하는지 전혀 모릅니다.** 그건 전부 훅 안에 있습니다.

---

# 🔍 OS마다 다른 권한 처리, 훅이 대신 맡는다

`ensurePermission`은 권한의 여러 상태를 받아 각각 다르게 처리하는 상태 기계입니다.

여기서 마음에 들었던 건, **OS를 직접 `if`로 나누지 않고 OS가 내려주는 권한 상태값으로만 분기**했다는 점입니다. 상태는 `granted`(허용), `notDetermined`(아직 안 물어봄), `denied`(거부), `permanentlyDenied`(영구 거부), `unknown`, `requesting`(요청 중) 으로 나뉩니다.

| 상태 | 처리 |
| --- | --- |
| granted | 통과 (거부 횟수 초기화) |
| permanentlyDenied | 즉시 설정 이동 안내 |
| notDetermined | 시스템 팝업 요청, 거부하면 재시도 없이 바로 설정 안내 |
| denied | 두 번까지 재요청, 두 번째에 설정 안내 |
| unknown | denied와 같은 로직 (두 번까지 재요청) |

처리 순서는 이렇습니다.

```text
ensurePermission(권한) 호출
        │
        ▼
   현재 권한 상태 확인
        │
        ├─ granted            → 통과 (거부 횟수 초기화)
        ├─ permanentlyDenied  → 설정 이동 안내 (재요청 없음)
        ├─ notDetermined      → 시스템 팝업 요청 → 거부 시 설정 안내
        └─ denied / unknown   → 두 번까지 재요청 → 2회째엔 설정 안내
```

OS에 따른 차이도 이 상태값에서 자연스럽게 갈렸습니다. `notDetermined`는 iOS에서 아직 한 번도 안 물어본 상태인데, 여기만 "거부하면 재시도 없이 바로 설정"으로 다릅니다. iOS는 한 번 거부하면 시스템 팝업을 다시 띄울 수 없기 때문입니다. 반면 `denied`와 `unknown`(주로 Android)은 둘 다 두 번까지 재요청하는 같은 로직을 탑니다.

거부 횟수는 전역 스토어에 권한별로 쌓아두고, 두 번에 도달하면 설정 이동 팝업을 띄웠습니다. 결국 호출하는 화면은 OS도 거부 횟수도 모른 채 권한 종류만 넘기면 됐고, "iOS냐 Android냐"를 `if`로 따로 나누지 않고 상태값으로 갈리게 한 것이 이 설계의 핵심이었습니다.

---

# 🔍 권한은 두 레이어에서 다뤄진다

사실 권한은 한 군데가 아니라 두 군데서 다뤄집니다. OS 권한 API는 네이티브만 만질 수 있어서, 실제 권한 조회·요청과 OS마다 다른 처리는 Flutter의 `PermissionService`가 맡습니다. WebView 안에서 도는 React 화면은 권한을 직접 만질 수 없으니, 그 서비스를 브릿지로 호출하는 `usePermissions` 훅으로 감쌌습니다. 재요청을 몇 번까지 할지 같은 UX 판단은 이 훅(웹)에 뒀습니다.

```text
[React 화면]
   │  hasPermission / ensurePermission 호출
   ▼
[usePermissions 훅 (웹)]    재요청 정책·설정 이동 판단
   │  브릿지 호출
   ▼
[PermissionService (Flutter)]    실제 OS 권한 조회·요청
   │   · iOS     : 네이티브로 권한 캐시 우회
   │   · Android : SDK 버전별 저장소 권한
   │   · 상태값 정규화
   ▼
[ iOS / Android OS ]
```

그래서 화면은 훅만 보면 되고, OS와 실제로 대화하는 일은 그 아래 네이티브가 맡는 구조입니다.

---

# 🔍 설정 갔다 돌아오면 자동으로 다시 맞춘다

권한을 거부하면 설정 앱으로 보내는데, 사용자가 거기서 권한을 켜고 돌아오는 "왕복"이 안 닫히면 화면이 옛 상태로 남습니다.

그래서 앱으로 복귀하면 Flutter가 최신 권한 상태를 다시 알려주고, 그걸 받아 전역 상태를 갱신하도록 했습니다. 특히 알림 권한이 바뀌면 서버의 알림 설정까지 함께 맞췄습니다.

---

# 🔍 알림만은 일부러 다르게 다뤘다

"권한 네 개를 똑같이 관리한다"고 하면 사실 반례가 있습니다. 알림은 예외로 뒀습니다.

- `ensurePermission('notification')`은 항상 `false`를 돌려주고, 설정 이동 팝업도 띄우지 않습니다.
- 대신 앱을 처음 켤 때 한 번만 요청하고, 허용되면 서버 알림 설정까지 켭니다.
- 거부 횟수 집계에서도 알림은 빠져 있습니다.

알림이 다른 권한과 성격이 달라서였습니다. 마이크·갤러리처럼 "기능을 쓰려는 순간 다시 요청"하는 흐름이 아니고, 재요청도 마이페이지 설정도 없는 권한이라, 같은 상태 기계에 억지로 넣기보다 따로 빼는 게 맞다고 봤습니다.

---

# 🧐 정리

권한 체크가 녹음·갤러리·앱 로드로 흩어져 있었고, 같은 권한이라도 iOS와 Android가 다르게 동작했습니다.

이걸 `usePermissions` 훅(웹)과 그 아래 `PermissionService`(Flutter)로 모아서, **호출하는 화면은 권한 종류만 말하면 되고, OS별 차이는 그 뒤에서 처리**되게 했습니다.

> 마이크·카메라·갤러리는 `ensurePermission` 상태 기계로, 알림은 앱 로드 1회 요청과 서버 동기화로.

앞 글이 마이크 권한 한 건의 예외 상황이었다면, 이 글은 그 경험을 권한 전반을 다루는 공통 구조로 끌어올린 셈입니다.
